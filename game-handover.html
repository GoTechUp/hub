Guidelines for handing over a Unity game project from developer to client

Game Handover Process

The game handover process is crucial to ensure a smooth transition from the development phase to the client, especially when using tools like Unity for development and PlasticSCM for version control. This page outlines the steps and documents necessary for a successful handover, including code, assets, and necessary instructions.

Handover Checklist

  • Ensure that all Unity project files are committed and up-to-date in PlasticSCM.
  • Check that all assets (textures, models, sounds, etc.) are properly organized and included in the handover.
  • Verify that all dependencies and third-party libraries (e.g., Unity Asset Store packages) are included or referenced correctly.
  • Create detailed documentation on the gameâ€™s functionality, structure, and mechanics.
  • Include any necessary credentials, API keys, or services information (e.g., game analytics, cloud storage, etc.).
  • Ensure the project is free of bugs or issues (if possible) and provide a final testing report.
  • Set up a version tag in PlasticSCM to mark the final handover version.

Step-by-Step Handover Guide

1. Unity Project Cleanup

The Unity project must be cleaned up before the handover. This involves:

  • Removing unnecessary assets or test scenes.
  • Ensuring all scenes are saved and properly named.
  • Verifying that all assets (models, textures, sounds) are correctly linked in the project.
  • Ensuring there are no missing or broken references.

2. Version Control via PlasticSCM

Before handing over the game, ensure the following in PlasticSCM:

  • Commit all changes, including scripts, scenes, assets, and configurations.
  • Tag the project with a version label (e.g., "Release_v1.0").
  • Ensure the repository is fully synchronized and there are no unresolved merge conflicts.
  • Provide the client with instructions on how to clone or sync the project repository from PlasticSCM.

3. Game Documentation

Documentation should cover the following areas:

  • Overview of the game mechanics and objectives.
  • Detailed descriptions of each game level and progression.
  • Control and input settings (e.g., keyboard, mouse, gamepad).
  • Third-party integrations (e.g., in-app purchases, advertising, social media sharing).
  • Instructions for any admin panels, APIs, or backend services used in the game.

4. Final Testing and Quality Assurance

Before the handover, it is essential to conduct a final round of testing to ensure the following:

  • The game runs smoothly on all target platforms (PC, Mac, Android, iOS, etc.).
  • All key features and mechanics are functioning as expected.
  • There are no major bugs or issues in the game (critical issues should be addressed before handover).
  • Provide the client with any known issues or areas for improvement, if applicable.

5. Packaging and Delivery

After final testing, the game should be packaged and delivered. Steps include:

  • Building the final version of the game for each platform.
  • Delivering the final Unity project folder, including all assets, scenes, and scripts.
  • Providing access to the PlasticSCM repository or sending the final build packages for each platform (e.g., .apk, .exe, .ipa).
  • Include any additional files needed for deployment (e.g., configuration files, settings, credentials).

6. Post-Handover Support

Provide the client with a support plan that includes:

  • Bug fixing and minor updates (if applicable).
  • Instructions for the client to make basic modifications or updates (e.g., changing game settings, adding content).
  • Contact information for any post-handover issues or emergencies.

The game handover process is critical for ensuring a successful release and transition from the developer to the client. By following these guidelines, both parties can ensure the game is fully ready for deployment and any future updates or improvements can be efficiently managed.
